# ADR-002: Управление плагинами Picodata

**Дата:** 2026-05-04
**Статус:** Принято

---

## Контекст

Picodata поддерживает плагины — динамически загружаемые расширения в виде `.so`-файлов.
Плагин устанавливается через DDL-команды и может открывать собственные сетевые порты.

Пример: плагин **Radix** реализует Redis-протокол поверх Picodata и открывает
отдельный порт для Redis-клиентов.

Плагины требуют управления жизненным циклом:
1. Регистрация (`CREATE PLUGIN`)
2. Задание migration context (параметры для первичной миграции)
3. Применение миграций (`ALTER PLUGIN MIGRATE TO`)
4. Привязка сервисов к тирам (`ALTER PLUGIN ADD SERVICE TO TIER`)
5. Включение (`ALTER PLUGIN ENABLE`)

Без оператора всё это делается вручную через admin-сокет или pgproto.
Цель — автоматизировать установку и обновление плагинов как часть reconcile-loop.

---

## Решение

### Где объявляются плагины

Плагины описываются на уровне тира в `spec.tiers[].plugins`:

```yaml
tiers:
  - name: default
    shareDir: /usr/share/picodata
    plugins:
      - name: radix
        version: "1.0.0"
        migrationContext:
          tier_for_db_0: "default"
          # ...
          tier_for_db_15: "default"
          unlogged: ""
        services:
          - name: radix
            listenerPort: 8082
```

Плагины привязаны к тиру, а не к кластеру — каждый тир может иметь независимый набор плагинов.

### Поле `shareDir`

Picodata ищет `.so`-файл плагина по пути `{shareDir}/{name}/{version}/`.
Поле `shareDir` задаётся на уровне тира и попадает в `instance.share_dir` в `config.yaml`.

**`share_dir` задаётся для каждого тира, включая тиры без плагинов.**
`CREATE PLUGIN` — кластерный DDL: при его выполнении Picodata обращается ко всем
инстансам кластера (в том числе к арбитру), и каждый инстанс должен уметь прочитать
директорию плагинов. Если `shareDir` не задан явно, оператор подставляет
`/usr/share/picodata` — стандартный путь, присутствующий во всех официальных образах.

### Когда запускается reconcile плагинов

Установка плагинов запускается **только когда все поды тира готовы** (`readyReplicas >= desiredReplicas`).
Это необходимо: DDL-команды `CREATE PLUGIN` и `ALTER PLUGIN MIGRATE` требуют
кворума кластера — часть из них выполняется на лидере и реплицируется на все инстансы.

### Как оператор подключается к Picodata

Оператор подключается через **pgproto (PostgreSQL-протокол)** к ClusterIP-сервису тира.
Используется библиотека `pgx/v5`. Подключение выполняется от имени пользователя `admin`
с паролем из Secret, указанного в `spec.adminPassword`.

Альтернатива — admin-сокет (`picodata admin`) — отклонена: требует exec в под,
что усложняет оператор и повышает требования к RBAC.

### Жизненный цикл плагина

```
CREATE PLUGIN {name} {version}
    ↓
ALTER PLUGIN {name} {version} ADD SERVICE {svc} TO TIER {tier}  (из spec.services, до миграции)
    ↓
ALTER PLUGIN {name} {version} SET migration_context.{key}='{value}'  (для каждого ключа)
    ↓
ALTER PLUGIN {name} MIGRATE TO {version}
    ↓
ALTER PLUGIN {name} {version} ADD SERVICE {svc} TO TIER {tier}  (из _pico_service, идемпотентно)
    ↓
ALTER PLUGIN {name} {version} ENABLE
```

> **Почему ADD SERVICE идёт до SET migration_context:** некоторые плагины (в частности Radix)
> валидируют параметры migration context и проверяют, что сервис уже привязан к тиру,
> указанному в `tier_for_db_N`. Если привязки ещё нет — команда SET возвращает ошибку.
> Поэтому ADD SERVICE выполняется дважды: первый раз — перед migration context (из spec),
> второй — после MIGRATE (из `_pico_service`, для сервисов не перечисленных в spec).

Шаги идемпотентны: повторное выполнение не приводит к ошибке.
Оператор проверяет текущее состояние через `_pico_plugin` и `_pico_service`
перед выполнением каждого шага.

### Зачем нужна таблица `_pico_service`

После `MIGRATE TO` Picodata регистрирует сервисы плагина в системной таблице `_pico_service`.
Оператор запрашивает её чтобы узнать список сервисов и к каким тирам они уже привязаны:

```sql
SELECT name, tiers FROM _pico_service
WHERE plugin_name = 'radix' AND version = '1.0.0'
```

Это позволяет:

1. **Не дублировать имена сервисов в CRD.** Имена сервисов определяются манифестом плагина,
   а не пользователем. После миграции оператор читает их из `_pico_service` и вызывает
   `ADD SERVICE {svc} TO TIER {tier}` для каждого. Пользователю не нужно перечислять
   сервисы в spec — только те, для которых нужен `listenerPort` (порт в config.yaml и Service).

2. **Идемпотентная привязка.** Поле `tiers` в `_pico_service` содержит список тиров,
   к которым сервис уже привязан. Оператор пропускает `ADD SERVICE TO TIER` если тир
   уже присутствует в этом списке — повторные reconcile-итерации безопасны.

Таблица `_pico_service` — источник правды о состоянии привязок. Поле `services`
в `PluginSpec` используется только для конфигурации listener-порта в `config.yaml`
и ClusterIP Service, а не для управления привязками.

**migration context** применяется всегда пока плагин не включён (`enabled = false`),
а не только при первичном создании. Это гарантирует корректное восстановление
после прерванного reconcile (например, перезапуска оператора между CREATE и MIGRATE).

### Migration context

Некоторые плагины требуют параметры для первичной миграции — например, Radix требует
указать тир для каждой из 16 Redis-баз данных и режим WAL:

```yaml
migrationContext:
  tier_for_db_0: "default"
  ...
  tier_for_db_15: "default"
  unlogged: ""   # "" = WAL включён, "UNLOGGED" = WAL отключён
```

Параметры передаются командой:
```sql
ALTER PLUGIN radix 1.0.0 SET migration_context.tier_for_db_0='default' OPTION(TIMEOUT=1200)
```

Плагины без migration context работают без дополнительных параметров — оператор
просто пропускает этот шаг (итерация по пустому map).

### Порты плагинов

Каждый сервис плагина с `listenerPort` получает:

1. **Запись в `config.yaml`** в секции `instance.plugin.{name}.service.{svc}.listener`:
   ```yaml
   instance:
     plugin:
       radix:
         service:
           radix:
             listener:
               enabled: true
               listen: "0.0.0.0:8082"
   ```
   Это заставляет Picodata открыть порт при старте инстанса.

2. **Порт в ClusterIP-сервисе** тира — клиенты могут обращаться к плагину
   через Kubernetes Service без port-forward.

В Kubernetes используется фиксированный порт вместо динамически вычисляемого
(`{{ instance_id | plus: N }}` из документации Picodata) — в контейнере
всегда один инстанс, поэтому динамика не нужна.

### Обновление плагина

При изменении `version` в spec оператор:
1. Устанавливает новую версию по полному жизненному циклу
2. Удаляет старые версии: `ALTER PLUGIN DISABLE` → `DROP PLUGIN`

Picodata поддерживает одновременно не более двух версий одного плагина,
поэтому оператор не накапливает старые версии.

### Переменная `PICODATA_PG_LISTEN`

Docker-образы Picodata могут содержать `ENV PICODATA_PG_LISTEN` с нестандартным портом.
Оператор явно выставляет эту переменную в pod template (`0.0.0.0:{pgPort}`),
чтобы значение из образа не перекрывало конфигурацию.

## Рассмотренные альтернативы

### Job/InitContainer для установки плагинов

Запускать отдельный Job после деплоя кластера.

Отклонено: Job одноразовый — не реагирует на изменение spec (обновление версии плагина).
Reconcile-loop оператора обрабатывает и первичную установку, и обновление однородно.

### Хранить состояние установки в status CR

Добавить в `status` поля `phase: Created | Migrated | Enabled` для каждого плагина.

Отклонено: нарушает принцип оператора — источник правды должен быть один
(реальное состояние Picodata). Два источника правды расходятся при ручном вмешательстве.
Текущий подход (всегда читать состояние из `_pico_plugin`) самодостаточен.

### Exec в под через kubectl

Выполнять команды через `kubectl exec` и admin-сокет.

Отклонено: требует монтирования сокета или наличия `kubectl` в образе оператора.
pgproto — стандартный интерфейс Picodata, доступен сетевым образом без привилегий на exec.

### Общий том (RWX PVC или hostPath) для бинарников плагинов

Монтировать один том со всеми `.so`-файлами сразу во все поды тира — вместо того чтобы включать плагин в образ контейнера.

Рассматривались два варианта:

**ReadWriteMany PVC** (NFS, CephFS, Longhorn). Все поды тира монтируют один том.  
Отклонено: StorageClass с поддержкой RWX недоступен в стандартных Kubernetes-дистрибутивах
(minikube, kubeadm bare-metal) без дополнительной настройки инфраструктуры.
Единый том становится единой точкой отказа: при его недоступности все поды тира
теряют доступ к плагину одновременно.

**hostPath** (директория на узле). Прямое монтирование пути с файловой системы узла.  
Отклонено: нарушает переносимость подов — файлы должны быть предварительно разложены
на каждом узле вручную. Оператор не может гарантировать наличие нужной версии плагина
на произвольном узле после перепланирования пода.

**InitContainer** как более проработанная альтернатива. InitContainer копирует `.so` из
отдельного образа плагина в `emptyDir`, который монтируется в основной контейнер.
Не требует RWX и переносим между узлами. Отложено как возможное улучшение:
потребует нового поля `pluginImage` в `PluginSpec` и генерации initContainer в StatefulSet.
Текущий подход (плагин в образе Picodata) проще в эксплуатации и не требует
поддержки двух образов на версию плагина.

---

## Последствия

- Установка и обновление плагинов полностью декларативны — описываются в CR
- Оператор идемпотентно приводит состояние плагинов к желаемому при каждом reconcile
- При прерванной установке оператор продолжит с места остановки (не с начала)
- Порты плагинов автоматически добавляются в Service и config.yaml
- Плагины без migration context поддерживаются без изменений в коде
- TLS для плагинов (listener.tls) запланирован в следующей итерации
